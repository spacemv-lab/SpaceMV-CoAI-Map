# map-ai 需求梳理

## 1. 背景

`map-ai` 现在已经有三块基础能力，但主链路还没有串起来：

- `map-core`：负责地图渲染、图层编辑、视角和交互状态管理。
- `GIS-DataManger`：已经有数据集、版本、上传解析、GeoJSON/MVT 接口。
- `map-ai web/server`：有页面壳和 API 容器，但账号、项目空间、权限还没落地。

当前最紧急的问题不是“把更多地图功能做出来”，而是先把下面这条链路打通：

`用户登录 -> 进入自己可访问的项目空间 -> 管理项目空间中的 GIS 数据 -> 在 map-core 中打开并保存项目空间状态`

这条链路打通后，后续再扩展协作、分享、AI 工作流、项目空间模板，成本才可控。

## 2. 产品目标

### 2.1 P0 目标

先完成一个可用的账号与数据管理系统，满足下面 4 件事：

1. 用户可以注册/登录，服务端知道“当前是谁”。
2. 用户可以看到自己有权限访问的项目空间。
3. 用户可以在项目空间下上传、查看、删除、加载 GIS 数据。
4. `map-core` 可以读取并保存某个项目空间的“当前状态”。

### 2.2 P1 目标

- 项目空间成员协作。
- 项目空间快照/历史版本。
- 分享链接、只读访问。
- AI 操作和项目空间/数据权限联动。

## 3. 核心概念

### 3.1 账号（Account/User）

表示真实登录用户，是权限和审计的起点。

需要记录：

- `id`
- `name`
- `email` 或 `phone`
- `passwordHash`
- `status`：正常、禁用
- `lastLoginAt`
- `createdAt` / `updatedAt`

说明：

- P0 先用“账号密码 + JWT/Session”即可，不必一开始做复杂 SSO。
- 后续如需接企业微信、微信扫码、统一认证，再通过 `UserIdentity` 扩展。

### 3.2 项目空间（Project Space）

`map-core` 渲染的不是“临时地图页面”，而是一个项目空间。它是后续业务的一级容器。

一个项目空间至少包含：

- 基本信息：名称、描述、封面、创建人
- 成员关系：谁能访问、谁能编辑
- 数据资源：该空间绑定的数据集
- 地图状态：当前加载哪些图层、样式、顺序、视角、时间轴、选中状态
- 历史快照：用于回滚和审计

结合现有 schema，`GIS-DataManger` 里的 `Project` 本质上已经接近这个概念。P0 不建议再造一个平行实体，建议直接把现有 `Project` 升级为“项目空间”。

### 3.3 数据集（Dataset）

数据集是项目空间下的资源对象，不应该脱离项目空间单独存在。

已具备的能力：

- `Dataset`
- `DatasetVersion`
- `GisFeature`
- `ValidationReport`
- `MappingProfile`

P0 重点不是重做数据模型，而是补齐：

- 数据归属到项目空间
- 数据操作归属到登录用户
- 数据访问受项目空间成员权限控制

### 3.4 项目空间状态（Project State）

项目空间状态是 `map-core` 当前 Zustand store 的可持久化版本。

核心内容来自 [map-state.ts](/home/xjw-linux/projects/TXWX-Monorepo/libs/features/map-core/src/lib/types/map-state.ts)：

- `viewport`
- `basemap`
- `layers`
- `activeLayerId`
- `stylePanel`
- `attributePanel`
- `popups`
- `selectedFeatureIds`
- `interaction`
- `selection`
- `hover`

但不是所有字段都适合直接落库。P0 先保存“可恢复画面”的最小集合：

- 底图
- 视角
- 图层列表与顺序
- 图层显隐
- 图层样式
- 图层绑定的数据集 ID / 版本 ID

先不做或延后：

- 临时 hover
- 临时 popup
- 会话态 AI 对话上下文
- 非必要的 UI 开关

## 4. 业务关系

P0 建议采用下面的关系模型：

- 一个 `User` 可以创建多个 `Project`
- 一个 `Project` 可以有多个 `ProjectMember`
- 一个 `Project` 可以有多个 `Dataset`
- 一个 `Dataset` 可以有多个 `DatasetVersion`
- 一个 `Project` 有一个“当前状态” `ProjectState`
- 一个 `Project` 可以有多个“历史快照” `ProjectSnapshot`

关系上要明确两件事：

1. 数据权限不直接挂在数据集上，而是通过项目空间成员权限继承。
2. `map-core` 加载的数据图层，必须能追溯到 `datasetId` 或 `datasetVersionId`。

## 5. 账号系统需求

### 5.1 P0 范围

- 登录
- 退出登录
- 获取当前用户信息
- 登录态校验
- 基于登录用户过滤项目空间和数据

### 5.2 不在 P0 的内容

- 第三方 OAuth
- 多组织体系
- 细粒度字段级权限
- 审批流

### 5.3 权限模型

P0 先做项目空间级角色，不做更细：

- `OWNER`：项目空间所有者，可管理成员、数据、空间状态
- `EDITOR`：可上传/删除数据，可保存项目空间状态
- `VIEWER`：只读查看、加载地图

至少需要限制：

- 未登录用户不能访问数据管理接口
- 非成员不能访问项目空间及其数据
- `VIEWER` 不能上传、删除数据
- 只有 `OWNER/EDITOR` 能保存项目空间状态

## 6. 数据管理需求

### 6.1 P0 必做

- 项目空间下的数据集列表
- 上传 GeoJSON / Shapefile / KML / CSV / XLSX
- 查看上传进度和解析状态
- 查看校验报告
- 删除数据集
- 加载数据集到地图
- 记录上传人、上传时间、版本号

### 6.2 数据管理与账号绑定要求

每一次数据动作都要知道“是谁做的”：

- 创建项目空间：`ownerId`
- 上传版本：`uploadedBy`
- 删除数据集：`deletedBy` 或操作日志
- 保存项目空间状态：`updatedBy`

前端不能再使用硬编码 `default-project-id`，必须改成：

- 从登录态获取当前用户
- 获取该用户可访问的项目空间列表
- 当前页面绑定选中的 `projectId`

### 6.3 与 map-core 的关系

数据管理不是独立后台页，而是项目空间的一部分。

至少需要支持两个动作：

1. 在数据管理页上传并管理数据。
2. 在项目空间中把某个数据集加载为地图图层。

因此 `Dataset` 返回给前端时，除了现有字段，还应包含：

- `projectId`
- `currentVersionId`
- `bbox`
- `geometryType`
- 可直接用于地图加载的资源信息：
  - 小数据量：GeoJSON URL
  - 大数据量：MVT URL

## 7. 项目空间需求

### 7.1 P0 必做

- 创建项目空间
- 打开项目空间
- 保存当前项目空间状态
- 最近访问列表

### 7.2 保存内容

项目空间保存时，至少要记录：

- 项目空间基础信息
- 当前地图状态 JSON
- 关联的数据图层引用
- 最近一次保存时间
- 保存人

### 7.3 快照策略

P0 采用“双层保存”：

- `Project.currentState`：当前最新状态，覆盖写
- `ProjectSnapshot`：按需保存的历史快照

这样可以同时满足“自动保存”和“历史回滚”。

基于现有 `libs/features/GIS-DataManger/prisma/schema.prisma`，建议新增或调整：

### 8.1 新增 `User`

用于登录和审计。

### 8.2 新增 `ProjectMember`

字段建议：

- `id`
- `projectId`
- `userId`
- `role`
- `invitedBy`
- `createdAt`

### 8.3 为 `Project` 增补项目空间属性

建议增加：

- `cover`
- `isArchived`
- `lastOpenedAt`
- `currentStateId`

### 8.4 新增 `ProjectState`

字段建议：

- `id`
- `projectId`
- `stateJson`
- `version`
- `updatedBy`
- `createdAt`
- `updatedAt`

### 8.5 新增 `ProjectSnapshot`

字段建议：

- `id`
- `projectId`
- `name`
- `stateJson`
- `createdBy`
- `createdAt`

### 8.6 为现有数据模型补审计字段

建议补充：

- `Dataset.createdBy`
- `Dataset.updatedBy`
- `Dataset.deletedAt`
- `Dataset.deletedBy`

现有 `DatasetVersion.uploadedBy` 已有，但现在写死为 `system`，需要切到真实用户。

## 9. API 边界建议

P0 接口建议按三组拆：

### 9.1 认证接口

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`

### 9.2 项目空间接口

- `GET /api/projects`
- `POST /api/projects`
- `GET /api/projects/:id`
- `GET /api/projects/:id/state`
- `PUT /api/projects/:id/state`
- `POST /api/projects/:id/snapshots`
- `GET /api/projects/:id/snapshots`

### 9.3 数据接口

建议把现有 `datasets` 接口语义改成项目空间视角：

- `GET /api/projects/:projectId/datasets`
- `POST /api/projects/:projectId/datasets/upload`
- `DELETE /api/projects/:projectId/datasets/:datasetId`
- `GET /api/projects/:projectId/datasets/:datasetId`
- `GET /api/projects/:projectId/datasets/:datasetId/geojson`
- `GET /api/projects/:projectId/datasets/:datasetId/mvt/:z/:x/:y`

是否保留现有 `/api/datasets/*` 可以看兼容策略，但新代码建议走项目空间路由，权限更清晰。

## 11. 实施优先级

### Phase 1：账号与项目空间骨架

- 新增用户表、项目成员表、项目状态表
- 实现登录、`me`、登出
- 实现项目空间列表和创建
- 前端建立登录态与项目空间上下文

### Phase 2：数据管理接入登录态

- 上传、删除、查询接口全部接入鉴权
- 数据按项目空间过滤
- `uploadedBy`、`ownerId` 改为真实用户
- 前端数据管理页面切换到当前项目空间

### Phase 3：项目空间状态保存

- 从 `map-core` 导出可持久化状态
- 实现自动保存/手动保存
- 打开项目空间时恢复图层、样式和视角

## 12. 当前结论

这次需求不应该理解成“单独补一个登录框”，而是要先建立一个真正的业务主模型：

- 账号是身份入口
- 项目空间是业务容器
- GIS 数据是项目空间资源
- `map-core` 状态是项目空间内容

最紧急的 P0 是：

1. 登录态建立
2. 项目空间列表与切换
3. 数据管理绑定项目空间和用户
4. 保存项目空间当前地图状态

只有这四个点落地后，`map-ai` 才具备后续扩展“协作空间”的基础。
