# 实时编辑功能技术调研与实现方案 v2

## 1. 目标重述

本功能不是单纯的“前端几何拖拽”，而是一个适配 **Nx 单体化全栈仓库** 的地图编辑能力，目标包含四层：

1. **本地交互实时反馈**
   - 点拖拽
   - 线/面顶点拖拽
   - 插入/删除节点
   - 吸附预览
2. **前端状态可恢复**
   - 编辑开始 / 结束
   - 撤销 / 重做
   - 草稿态 / 已提交态
3. **服务端可持久化**
   - 编辑结果可保存到数据库
   - 保存失败可回滚或重试
4. **多端实时同步可扩展**
   - 至少预留编辑事件广播、版本校验、冲突处理能力
   - 首期可先支持“单用户编辑 + 多端只读观察”

结论先行：  
本项目应采用 **“共享编辑契约 + 服务端编辑会话 + map-core 前端交互适配”** 的方案，而不是把实时编辑全部压进 `map-core` 的本地 store 和 renderer。

---

## 2. 为什么要升级原方案

原方案最大的问题不是交互细节，而是架构层级过低。它更像“Cesium 交互实现草稿”，但当前仓库是一个 Nx 单体化全栈仓库，技术方案必须天然回答下面几个问题：

- 编辑命令由谁定义，前后端如何共享类型？
- 几何修改是直接改前端 store，还是走统一 mutation pipeline？
- 保存失败怎么办？版本冲突怎么办？
- 将来多人协同是推翻重来，还是沿当前设计自然演进？
- Agent / AI 是否能感知和消费编辑事件？

如果这些问题不先定义，前端交互写得越快，后面重构成本越高。

---

## 3. 设计原则

### 3.1 单体仓库优先原则

既然已经是 Nx monorepo，就应该利用 monorepo 的优势：

- **共享类型**
  - 编辑命令 DTO
  - 编辑事件 DTO
  - 几何补丁结构
  - 版本元数据
- **共享验证**
  - 前后端使用同一套 schema
  - 避免前端先做一套、后端再重写一套
- **共享测试夹具**
  - GeoJSON 样例
  - 编辑事件回放样例
  - 冲突场景样例

### 3.2 前端交互与服务端协议解耦

`map-core` 负责：

- 鼠标命中
- 顶点控制点渲染
- 吸附候选高亮
- 本地草稿预览
- 画笔工具约束

`server` 负责：

- 编辑会话
- 权限校验
- 数据版本校验
- 持久化
- 广播

### 3.3 “实时”按阶段定义

首期不要直接承诺完整多人协同。推荐分三阶段：

1. **Phase A**
   - 单用户本地实时编辑
   - 保存到后端
2. **Phase B**
   - 编辑事件实时广播
   - 其他客户端看到结果更新
3. **Phase C**
   - 多人同时编辑
   - 版本冲突检测
   - 简化锁或 merge 策略

## 4. 推荐总体架构

### 4.1 分层

```text
apps / libs
├── libs/shared/map-edit-contract
│   ├── 编辑命令 DTO
│   ├── 编辑事件 DTO
│   ├── geometry patch 类型
│   └── schema / validator
│
├── libs/features/map-core
│   ├── Cesium 交互适配
│   ├── 顶点/吸附/幽灵渲染
│   ├── 编辑草稿 store
│   └── 命令发送器
│
├── apps/map-ai/server 或独立 feature/server lib
│   ├── edit gateway / controller
│   ├── edit session service
│   ├── patch apply service
│   ├── version check
│   └── persistence adapter
│
└── web app
    ├── 权限与页面会话
    └── 实时编辑 UI 组合
```

### 4.2 运行时链路

```text
用户拖动顶点
-> map-core 生成 draft patch
-> 本地实时预览
-> 用户确认/自动提交
-> emit EditCommand
-> server 校验 session / version / permission
-> apply patch
-> 持久化
-> 广播 EditEvent
-> 当前端与其他端同步更新
```

### 4.3 为什么这是更适合本仓库的架构

- 充分利用 Nx 的共享代码能力
- 不把后端语义硬编码到 Cesium renderer
- 对 AI/Agent 更友好，后续可直接消费编辑事件
- 将来加协同编辑时，不需要推翻前端交互层

## 5. 数据与契约设计

### 5.1 推荐新增共享库

建议新增：

`libs/shared/map-edit-contract`

职责：

- 定义编辑命令
- 定义编辑事件
- 定义 patch 结构
- 定义版本对象
- 提供 validator

### 5.2 命令模型

不要把“编辑”建模成直接改 Zustand。  
应该建模成明确命令。

```ts
export type EditCommand =
  | MovePointCommand
  | MoveVertexCommand
  | InsertVertexCommand
  | DeleteVertexCommand
  | MoveGeometryCommand
  | CommitDraftCommand
  | CancelDraftCommand;

export interface BaseEditCommand {
  commandId: string;
  conversationId?: string;
  sessionId: string;
  layerId: string;
  featureId: string;
  clientId: string;
  baseVersion: number;
  issuedAt: number;
}
```

关键点：

- `commandId`
  - 用于幂等
- `baseVersion`
  - 用于冲突检测
- `sessionId`
  - 用于区分编辑会话
- `clientId`
  - 用于多端识别来源

### 5.3 事件模型

```ts
export type EditEvent =
  | EditDraftUpdatedEvent
  | EditCommittedEvent
  | EditRejectedEvent
  | EditPresenceEvent;
```

至少需要三类服务端事件：

- `draft_updated`
  - 可选，用于共享草稿
- `committed`
  - 保存成功后的正式结果
- `rejected`
  - 版本冲突、权限失败、几何非法

### 5.4 Patch 模型

对 geometry 编辑，优先传 **patch**，不要每次整条 feature 全量覆盖。

```ts
export type GeometryPatch =
  | {
      type: 'move_vertex';
      vertexIndex: number;
      position: [number, number];
    }
  | {
      type: 'insert_vertex';
      segmentIndex: number;
      position: [number, number];
    }
  | {
      type: 'delete_vertex';
      vertexIndex: number;
    };
```

这样做的好处：

- 网络负载更小
- undo/redo 可直接复用
- 更容易做冲突分析
- 更适合回放和审计

## 6. 前端架构设计

### 6.1 不新增第二套“编辑图层真相源”

当前 `map-core` 已有：

- `activeLayerId`
- `selection`
- `interaction.mode`

因此不建议再引入一个独立的 `editing.activeLayerId` 去和现有状态并存。  
更合理的方式是：

- `activeLayerId`
  - 继续表示当前编辑归属图层
- 新增 `editSession`
  - 只表示编辑过程中的草稿、拖拽、吸附、历史记录

推荐结构：

```ts
export interface EditSessionState {
  status: 'idle' | 'editing' | 'saving' | 'error';
  editedFeatureId: string | null;
  draggedVertex: {
    featureId: string;
    vertexIndex: number;
    originalPosition: [number, number];
  } | null;
  snapCandidate: {
    featureId: string;
    segmentIndex: number;
    position: [number, number];
    screenDistance: number;
  } | null;
  draftPatches: GeometryPatch[];
  undoStack: GeometryPatch[];
  redoStack: GeometryPatch[];
  baseVersionByFeature: Record<string, number>;
}
```

### 6.2 map-core 内部职责拆分

推荐模块划分：

#### A. `edit-session-store.ts`

职责：

- 草稿 patch
- 撤销/重做
- 拖拽状态
- 吸附候选

补充说明：

- 只保存前端临时编辑过程状态
- 不承担最终持久化真相源
- commit 成功后应以服务端回写结果为准

#### B. `vertex-overlay-renderer.tsx`

职责：

- 渲染编辑控制点
- 渲染 hover / active vertex
- 渲染 segment hover

建议：

- 默认仅渲染当前编辑图层的控制点
- 默认仅对线 / 面要素启用
- 节点很多时优先考虑轻量 primitive，而不是无上限使用 Entity

#### C. `edit-interaction-manager.tsx`

职责：

- Cesium 事件绑定
- 命中测试
- 生成 patch
- 更新本地 draft

细化职责：

- `LEFT_DOWN`
  - 识别是否点中了 vertex / feature / segment
- `MOUSE_MOVE`
  - 拖拽中的 patch 实时生成
  - 吸附候选更新
- `LEFT_UP`
  - 结束拖拽
  - 将本次连续操作聚合成单条 undo 记录
- `RIGHT_CLICK` / `Delete`
  - 节点删除

#### D. `edit-command-bridge.ts`

职责：

- 把本地 patch 提交为命令
- 对接 socket / http
- 处理 ack / reject / commit

#### E. `geometry-constraint.ts`

职责：

- 根据当前编辑图层几何类型约束可用工具
- 防止点图层进入线/面绘制，反之亦然
- 在 UI 与命令提交两侧做双重保护

建议规则：

- `Point` 图层
  - 允许：选择、点编辑、点绘制
  - 禁止：线绘制、面绘制
- `LineString` 图层
  - 允许：选择、线编辑、线绘制
  - 禁止：点绘制、面绘制
- `Polygon` 图层
  - 允许：选择、面编辑、面绘制
  - 禁止：点绘制、线绘制

### 6.3 为什么不建议把一切塞到一个 manager

原始草案倾向于把事件、相机、吸附、store 操作都堆进一个组件。  
这会带来三个问题：

- 很难测试
- 很难复用
- 很难接服务端协议

对于当前仓库，更应该把“交互层”和“同步层”分开。

## 7. 吸附技术方案

### 7.1 原则

Turf 可以用，但不能把它当成吸附主框架。  
对于 Cesium 编辑，推荐策略是：

- **屏幕空间判定为主**
- **几何计算为辅**

### 7.2 推荐算法

1. 从鼠标位置拿到屏幕坐标
2. 从当前编辑要素及附近候选要素中收集 segment
3. 把候选 segment 的端点或采样点投影到屏幕
4. 直接计算鼠标到屏幕线段的像素距离
5. 若命中阈值，再使用 Turf / 几何工具算精确落点

补充候选筛选建议：

- 首期只检测当前编辑 feature 的 segment
- 第二阶段再扩展到当前编辑图层其他 feature
- 最后再扩展到跨图层吸附

这样可以把吸附检测的性能风险控制在可接受范围内

### 7.3 为什么这样更稳

- 吸附阈值本来就是像素语义
- 视角变化时体验更一致
- 不依赖脆弱的“米转像素”估算

### 7.4 Turf 的合理角色

Turf 适合做：

- 最近点计算
- segment 上的插值点
- 自交检测
- 几何合法性校验

Turf 不适合单独承担：

- Cesium 屏幕命中
- 视角相关的像素容差判断

### 7.5 可行性结论

`@turf/turf` 可以引，但应当作为 **几何工具库**，不是编辑架构核心。

## 8. 实时同步与后端设计

### 8.1 首期建议

首期做：

- 单用户编辑
- 保存命令提交到服务端
- 服务端返回 commit 结果
- 其他客户端收到 committed 事件并刷新

先不要做：

- 多用户同时拖同一要素
- CRDT
- 自动 merge

### 8.2 服务端最小能力

需要服务端提供：

#### A. Edit Session Service

- 创建编辑会话
- 校验用户是否可编辑
- 管理 session ttl

#### B. Edit Command Handler

- 校验命令 schema
- 校验 `baseVersion`
- 应用 patch
- 生成新版本

#### C. Persistence Adapter

- 把 geometry / attributes 写回数据库
- 失败时返回明确错误

#### D. Gateway / Broadcast

- 对当前会话广播 committed / rejected 事件

### 8.2.1 前端与服务端的职责边界

前端可直接完成：

- 顶点 hover / active 样式
- 拖拽幽灵效果
- 本地 undo/redo
- 几何类型工具约束

必须由服务端最终裁决：

- feature 是否可编辑
- `baseVersion` 是否过期
- patch 是否可应用
- 数据是否成功持久化

### 8.3 冲突策略

首期建议最简单的版本锁：

- 每个 feature 维护 `version`
- 客户端提交时附带 `baseVersion`
- 服务端不匹配则拒绝
- 前端提示“数据已被他人更新，请刷新后重试”

这比直接上 CRDT 更现实，也更适合当前阶段。

## 9. 前沿性判断

### 9.1 哪些不是前沿

下面这些不应被包装成“前沿方案”：

- 纯前端 Zustand 编辑状态
- Turf + Cesium 的局部吸附工具
- renderer 里直接改 entity
- 本地 undo/redo

这些都属于成熟工程实现，不是前沿。

### 9.2 本项目真正值得追求的前沿点

如果要体现前沿性，更值得投入的是：

1. **命令化编辑管线**
   - geometry / attribute 共用 mutation pipeline
2. **可回放编辑事件**
   - 支持调试、审计、AI 消费
3. **共享契约驱动**
   - web / server / agent 同用一套 edit contract
4. **渐进式协同能力**
   - 先版本锁，再细化冲突模型

### 9.3 结论

这个项目最合理的“前沿性”不是追逐复杂图形算法，而是把地图编辑做成一个 **全链路、可协同、可回放、可被 AI 理解的编辑系统**。

## 10. 可行性评估

### 10.1 高可行

- 点拖拽编辑
- 线面顶点编辑
- 屏幕空间吸附
- 本地撤销/重做
- 单 feature 版本锁
- socket 广播 committed 结果
- DrawToolbar 几何类型约束
- 顶点 overlay 独立渲染

### 10.2 中等可行

- 多图层联合吸附
- 面整体拖动
- 属性编辑与几何编辑统一提交
- 草稿态共享
- 视口内 segment 裁剪优化
- 空间索引加速吸附检测

### 10.3 暂不建议首期投入

- CRDT/OT 级多人协同
- 离线编辑合并
- 完整 topology editing
- 复杂 3D 地形贴合编辑

## 11. 分阶段实施路径

### Phase 1: 本地编辑内核

目标：

- 点、线、面基础编辑
- 本地草稿和 undo/redo
- 顶点 overlay
- 几何类型约束

落点：

- `map-core` 内新增 edit session store
- 新增 vertex overlay renderer
- 新增 edit interaction manager
- 新增 geometry constraint 模块并接入 `DrawToolbar`
- 新增基础 geometry utils（几何类型映射、geometry 校验、patch helper）

### Phase 2: 服务端保存

目标：

- 定义共享 edit contract
- 提交命令到服务端
- 持久化与版本校验

落点：

- 新增 shared contract lib
- server 新增 edit command handler
- server 增加 feature version 校验
- server 返回 committed / rejected 事件

### Phase 3: 实时同步

目标：

- committed 事件广播
- 其他端更新地图数据

落点：

- 复用现有 socket 基础设施
- map-core 增加 edit event bridge
- 当前编辑端处理 commit ack / reject rollback

### Phase 4: 协同增强

目标：

- presence
- feature-level lock 或软锁
- 冲突提示
- draft shared 预览（可选）

## 12. 性能优化建议

### 12.1 节点渲染

- 仅渲染当前编辑图层中的节点
- 仅渲染当前视口相关节点
- 节点数量大时优先使用轻量 primitive，而不是持续堆积 Entity

### 12.2 吸附检测

- 拖拽中优先检测局部 segment
- 使用 `requestAnimationFrame` 或固定频率节流 `MOUSE_MOVE`
- 当图层规模变大时再引入空间索引

### 12.3 编辑历史

- 连续拖拽过程只在结束时写入一条 undo 记录
- 限制 undo 栈大小，例如 50 步
- 避免每帧存完整 geometry 快照

## 13. 测试策略

### 13.1 单元测试

- geometry utils
- patch apply / reverse patch
- 吸附检测算法
- edit session store
- version conflict 判定

### 13.2 集成测试

- 开启编辑 -> 拖拽点 -> 生成 patch -> commit 成功
- 线节点插入 -> 撤销 -> 重做
- 几何类型约束：点图层时禁止线/面工具
- commit reject 后本地提示与恢复

### 13.3 E2E 测试

- 完整用户编辑保存流程
- 两个客户端同时观察，一个编辑、另一个接收 committed 更新
- 过期版本提交失败

## 14. 最终推荐方案

### 14.1 技术选型

- **Cesium 原生交互**：保留
- **Turf.js**：可引入，但仅作为几何工具
- **共享编辑契约库**：必须新增
- **Socket 实时广播**：复用现有基础设施并扩展
- **版本锁**：首期必须有

### 14.2 架构决策

采用：

**本地实时预览 + 共享编辑命令 + 服务端版本校验 + 事件广播**

不采用：

- 纯前端 store 直改作为最终方案
- 把实时编辑等同于本地拖拽
- 一开始就做重型协同算法

### 14.3 关键结论

对本仓库而言，正确方向不是“先把 Cesium 编辑交互写完再说”，而是：

1. 先定义共享编辑契约
2. 再做前端交互适配
3. 再接服务端持久化和广播
4. 最后渐进增强协同能力

这才符合 Nx 单体化全栈仓库的技术路线，也能避免后续大面积返工。
