# 地图交互架构设计

> 文档目标：总结当前地图交互实现，并给出后续扩展编辑、选择、hover、popup 的统一设计方向。

## 1. 设计目标

地图交互系统需要同时满足四类需求：

1. 浏览交互
   - hover 高亮
   - click 选择
   - popup 展示

2. 编辑交互
   - 点拖拽
   - 线面绘制
   - 后续顶点编辑

3. UI 联动
   - 图层管理器编辑态
   - 属性面板选中态
   - popup 跟随

4. 生命周期稳定
   - viewer 重建可恢复
   - 图层删除后交互状态自动清空
   - 临时高亮与正式渲染分离

## 2. 当前总体架构

```text
MapViewer
  ├─ MapContainer          -> 创建 Cesium Viewer
  ├─ LayerRenderer         -> 图层同步
  ├─ DrawRenderer          -> 绘制交互
  ├─ InteractionManager    -> hover / click / drag
  └─ PopupContainer        -> React 弹窗渲染

Zustand Store
  ├─ interaction.mode
  ├─ activeLayerId
  ├─ selection
  ├─ hover
  ├─ popups
  └─ layers
```

当前职责分工是：

- `LayerRenderer`: 图层与 Cesium 对象同步
- `InteractionManager`: 地图要素交互
- `DrawRenderer`: 绘图模式交互
- `PopupContainer`: 屏幕空间弹窗显示

这个边界比早期版本清晰得多，尤其是 hover 逻辑已经从 `LayerRenderer` 中拆出。

## 3. 状态模型

当前交互相关状态主要在 `use-map-store.ts`：

- `interaction.mode`
  - `default`
  - `draw_point`
  - `draw_line`
  - `draw_polygon`
  - `measure_distance`
  - `measure_area`
  - `select`

- `activeLayerId`
  - 当前编辑图层

- `selection`
  - `{ layerId, featureId, properties }`

- `hover`
  - `{ layerId, featureId }`

- `popups`
  - 弹窗数组，支持地图定位后跟随

### 当前设计意义

这套状态把“交互反馈”从 Cesium 对象抽离到了业务 store，带来两个收益：

1. UI 面板可以直接订阅交互结果
2. 图层删除时可以统一回收交互状态

## 4. 当前交互实现

## 4.1 Hover 高亮

当前 hover 由 `InteractionManager` 统一处理，分两条路径：

1. `Entity` 路径
   - 适用于线/面 DataSource
   - 命中后修改 entity 的临时高亮样式
   - 离开时恢复原始样式

2. `Primitive` 路径
   - 适用于点图层
   - 通过 point renderer registry 反查 `layerId + featureId`
   - 使用 `HoverOverlayManager` 创建临时 overlay
   - 离开时销毁 overlay

### 为什么点 hover 走 overlay

原因很明确：

1. 点不是 DataSource，而是自定义 renderer
2. 直接改原 primitive 容易污染原始样式
3. 多图层、大量点时，保存/恢复原样式复杂且脆弱
4. overlay 生命周期短，更符合 hover 的语义

因此当前推荐原则是：

- 线面可以临时改原 entity
- 点优先用 overlay

## 4.2 Click 选择

当前点击选择分成两块：

1. `LayerRenderer`
   - 负责 DataSource Entity 的点击拾取
   - 解析 `layerId / featureId / properties`
   - 写入 `selection`

2. `InteractionManager`
   - 负责点击后创建 popup

这说明当前 click 链路还没有完全收口，存在“选择”和“popup”分别归不同组件处理的现状。

### 当前问题

1. 线面点击与点点击还不完全统一
2. `LayerRenderer` 仍保留一部分交互职责
3. 交互架构已经在向统一收口演进，但还未彻底完成

## 4.3 Drag 编辑

当前已实现的是点要素拖拽，逻辑在 `InteractionManager`：

1. `LEFT_DOWN`
   - 命中 entity
   - 结合 `activeLayerId` 判断是否可编辑
   - 若命中点要素，则进入拖拽状态

2. `MOUSE_MOVE`
   - 若处于拖拽态，使用 `camera.getPickRay + globe.pick`
   - 实时更新 entity 位置

3. `LEFT_UP / LEFT_CLICK`
   - 将新坐标回写到 store 中的 GeoJSON

这个实现已经能支撑基础点编辑，但仍然属于“轻量版”：

1. 只覆盖点
2. 没有拖拽吸附
3. 没有编辑历史
4. 没有后端持久化

## 4.4 Draw 绘制

`DrawRenderer` 负责绘图模式：

1. 根据 `interaction.mode` 切换交互处理器
2. 通过临时 entity 预览线/面
3. 右键完成绘制
4. 将几何转成 GeoJSON Feature 回写到目标图层

当前设计特征：

1. 绘图与普通 hover/click 分开
2. 绘制期间使用临时 entity
3. 结束后才落库到 `LayerState.data`

这是合理的，因为“绘制中的形状”和“正式图层要素”不是同一生命周期对象。

## 4.5 Popup 跟随

`PopupContainer` 采用 React 组件 + `scene.postRender` 的模式：

1. popup 存的是地理坐标
2. 每帧用 `SceneTransforms.worldToWindowCoordinates`
3. 将地图坐标投影到屏幕坐标
4. 直接更新 DOM transform

这是当前比较务实的实现，优点是：

1. 不需要使用 Cesium 的默认 infoBox
2. popup 样式完全由 React 控制
3. 可以和业务 UI 风格统一

## 5. 当前架构中的关键设计原则

## 5.1 临时反馈与正式渲染分离

这是当前交互架构最重要的原则。

例子：

1. 点 hover 使用 overlay，而不是改原始点
2. 绘图过程使用临时 entity，而不是先写正式图层
3. popup 用 React UI 跟随，而不是把信息绑定到 Cesium 默认面板

好处：

1. 避免渲染状态和交互状态互相污染
2. 更容易做清理
3. 更容易扩展不同交互模式

## 5.2 交互命中结果要先标准化

交互层面对的是两类 Cesium 命中对象：

1. `pickedObject.id instanceof Cesium.Entity`
2. `pickedObject.primitive instanceof Cesium.PointPrimitive | Cesium.Billboard`

因此地图交互的本质不是“直接操作 Cesium 命中对象”，而是：

1. 先把命中对象解析为业务语义
2. 再更新 store
3. 再决定是否创建临时视觉反馈

建议后续统一抽象出：

```ts
type PickedFeature =
  | { kind: 'entity'; layerId: string; featureId: string; entity: Cesium.Entity }
  | { kind: 'primitive'; layerId: string; featureId: string; primitive: Cesium.PointPrimitive | Cesium.Billboard };
```

这样 hover、click、drag、popup 就可以共享一套解析入口。

## 5.3 store 是交互真相源，Cesium 只是投影

当前系统已经基本形成这个模式：

1. Cesium 对象承接即时视觉表现
2. store 承接业务真实状态
3. 删除图层时先清 store，再清资源
4. 编辑完成后再把几何回写 store

这对 AI 操作、面板联动、后续持久化都很重要。

## 6. 当前痛点

虽然已经能工作，但还存在几处不够整洁的地方：

1. 点击选择仍分散在 `LayerRenderer` 和 `InteractionManager`
2. 点 primitive 的 click 还没完全纳入统一选择链
3. `DrawRenderer` 与 `InteractionManager` 都在直接监听鼠标事件，未来可能出现模式冲突
4. 线/面编辑仍停留在设计阶段，未与现有交互内核统一
5. 编辑态、hover 态、selection 态的优先级还没有明确规范

## 7. 推荐演进方向

## 7.1 统一拾取内核

建议新增一个内部的 pick resolver：

1. 输入：`scene.pick()` 结果
2. 输出：标准化 `PickedFeature | null`
3. 调用者：
   - hover
   - click
   - drag
   - popup

收益：

1. 避免重复写命中判断
2. 线面与点可以统一
3. 后续编辑态过滤更容易做

## 7.2 明确交互优先级

建议后续定义优先级：

1. 拖拽中
2. 绘制中
3. 编辑节点中
4. hover
5. 普通 click

否则未来接入线面顶点编辑后，事件竞争会明显增加。

## 7.3 线面编辑作为 InteractionManager 的扩展，而不是新体系

现有调研文档里提到 `EditInteractionManager`、节点渲染、吸附引擎，这些都合理，但不建议再平行造一套完全独立的交互系统。

更合理的方式是：

1. 保持 `InteractionManager` 作为统一入口
2. 在线/面编辑模式下，挂接 vertex interaction 子模块
3. 节点、吸附、幽灵预览作为 overlay renderer

也就是：

- 一个交互内核
- 多种模式插件

而不是：

- 普通交互一套
- 编辑交互再来一套

## 7.4 完成选择与 popup 收口

建议下一步完成：

1. 把 `LayerRenderer` 中的点击选择迁到 `InteractionManager`
2. 点和线面共用选择更新逻辑
3. popup 只消费 `selection` 或统一 click 结果

这样交互职责会更干净：

- `LayerRenderer` 只管图层同步
- `InteractionManager` 只管交互
- `PopupContainer` 只管 UI 展示

## 8. 推荐的最终目标架构

```text
InteractionManager
  ├─ PickResolver
  ├─ HoverController
  ├─ SelectionController
  ├─ PopupController
  ├─ DragController
  └─ EditController
         ├─ VertexOverlayRenderer
         ├─ GhostOverlayRenderer
         └─ SnapEngine
```

其核心原则是：

1. 所有交互从统一拾取入口进入
2. 所有临时视觉反馈走 overlay
3. 所有业务结果写回 store
4. 所有 Cesium 对象都只是当前状态的投影

## 9. 结论

当前地图交互架构已经完成了从“渲染器里顺手写交互”到“独立交互管理器”的第一阶段收敛，尤其在点图层 hover 上已经验证了正确方向：

1. 图层同步与交互逻辑必须分离
2. 点图层需要单独的 primitive 交互通道
3. hover 这类短生命周期反馈应优先采用 overlay
4. 交互系统的下一步重点不是继续堆功能，而是统一拾取、统一状态、统一优先级

这份文档可以作为后续编辑、吸附、选择、popup 收口的设计基线。
