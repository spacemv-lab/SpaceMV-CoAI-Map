# 地图架构总览

> 适用对象：第一次接触 `map-core` 的开发者。
> 阅读顺序建议：
> 1. 本文
> 2. `10-图层渲染架构归档.md`
> 3. `11-地图交互架构设计.md`

## 1. 地图模块在做什么

`map-core` 是地图前端内核，负责三类能力：

1. 图层管理
   - 加载、删除、显隐、排序、样式设置

2. 地图交互
   - hover、选择、popup、拖拽、绘制

3. 状态承接
   - 用统一 store 连接 UI、Cesium 渲染、后续 AI 操作

## 2. 一句话理解当前架构

当前架构可以概括为：

- `Store` 管业务状态
- `LayerRenderer` 管图层同步
- `InteractionManager` 管地图交互
- `DrawRenderer` 管绘制流程
- `PopupContainer` 管 popup UI

点图层是一个单独子系统：

- 线/面走 `DataSource + Entity`
- 点走 `PointRendererManager + Primitives`

## 3. 最重要的设计边界

### 3.1 业务图层 != Cesium 原生对象

业务里有“图层”概念，但 Cesium 原生并没有。

所以当前做法是：

1. `LayerState` 作为业务图层定义
2. `LayerRenderer` 把它同步成：
   - `GeoJsonDataSource`
   - `Entity`
   - `PointPrimitiveCollection`
   - `BillboardCollection`

### 3.2 长生命周期渲染与短生命周期交互分离

这是现在最应该记住的原则。

- 长生命周期：
  - 正式图层
  - DataSource
  - Primitive renderer

- 短生命周期：
  - hover 高亮
  - 绘制预览
  - popup
  - 编辑态幽灵对象

结论：

- 渲染对象负责“长期存在”
- overlay / 临时 entity 负责“即时反馈”

## 4. 你会最常碰到的几个组件

### `use-map-store.ts`

地图模块状态中心，核心字段包括：

- `layers`
- `activeLayerId`
- `interaction.mode`
- `selection`
- `hover`
- `popups`

### `layer-renderer.tsx`

图层同步调度器，负责：

1. 加载数据
2. 创建/销毁 Cesium 渲染对象
3. 更新显隐和样式
4. 清理已删除图层的渲染资源

注意：

- 它不应该继续承载 hover 之类的交互逻辑

### `interaction-manager.tsx`

交互统一入口，负责：

1. hover
2. click
3. 点拖拽
4. popup

这是后续继续扩展编辑能力的主要落点。

### `draw-renderer.tsx`

绘制模式专用交互：

1. 点/线/面的绘制
2. 临时预览 entity
3. 完成后转成 GeoJSON 回写 store

### `point-renderer.ts` 及其子类

点图层子系统：

1. `PointRendererManager`
2. `PointPrimitiveRenderer`
3. `BillboardRenderer`

用途：

- 解决点图层在性能、表现力和交互上的特殊需求

## 5. 为什么点图层要单独处理

因为点图层和线面图层不是一类问题。

线/面：

- `GeoJsonDataSource + Entity` 基本够用

点：

- 样式变化多
- 可能数据量大
- 既要支持简单圆点，也要支持图片/符号
- hover / 选择 / 编辑链路和线面不同

所以当前结论已经很明确：

- 点图层单独做 renderer 是对的

## 6. 当前已经稳定的架构结论

### 图层渲染

当前稳定结论：

1. 线/面走 DataSource
2. 点走 Point Renderer
3. 图层删除必须同时清理 store 和 Cesium 资源

### 地图交互

当前稳定结论：

1. hover 逻辑归 `InteractionManager`
2. 点 hover 优先用 overlay，不改原 primitive
3. popup 走 React UI，而不是 Cesium 默认 infoBox

## 7. 当前还没完全收口的地方

还有几处是“能用，但还不够干净”的：

1. 点击选择还没有完全统一到 `InteractionManager`
2. 点 primitive 的 click 链还可以继续补齐
3. 线/面编辑还在设计和扩展阶段
4. 图层排序与 Cesium 实际渲染顺序仍需继续打磨

## 8. 建议新人怎么继续开发

如果你要继续开发，建议遵守以下规则：

1. 改图层同步逻辑，优先看 `layer-renderer.tsx`
2. 改 hover / click / drag，优先看 `interaction-manager.tsx`
3. 改绘制模式，优先看 `draw-renderer.tsx`
4. 改点图层渲染能力，优先看 `point-renderer.ts` 及子类
5. 不要把交互逻辑重新塞回渲染同步层

## 10. 最终总结

这个模块当前最重要的架构认知不是“Cesium 怎么用”，而是：

1. 业务状态和 Cesium 对象要分层
2. 图层同步和交互处理要分层
3. 点图层必须视为独立子系统
4. 临时反馈优先用 overlay，而不是污染正式渲染对象

只要守住这四条，后续继续做编辑、吸附、撤销重做，都不会再把结构带歪。
