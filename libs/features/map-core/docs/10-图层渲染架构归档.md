# 图层渲染架构归档

> 归档目的：将点图层样式问题分析、Cesium 图层管理设计、点图层渲染架构总结三份文档收敛为一份当前有效的设计归档。

## 1. 背景与问题演进

地图模块最初基于 `Cesium.GeoJsonDataSource + Entity` 实现统一渲染。这个方案适合线面图层，但在点图层上很快暴露出三个问题：

1. 点样式能力不足
   - `GeoJsonDataSource` 默认把点转成 `Billboard`
   - 颜色可改，但尺寸、单位、符号、图片、旋转等样式无法完整表达

2. 点数据性能与表现力矛盾
   - `Entity` 方案易用，但海量点性能差
   - `PointPrimitive` 性能好，但能力偏底层
   - `Billboard` 表现力强，但交互和管理需要单独设计

3. Cesium 原生没有“业务图层”概念
   - 应用侧需要图层树、显隐、删除、重排序、样式设置
   - Cesium 侧只有 `DataSource`、`Entity`、`Primitive`

因此最终形成了“线面走 DataSource，点走独立 Renderer”的双轨架构。

## 2. 当前总体架构

```text
React UI
  ├─ LayerManager
  ├─ StylePanel
  └─ MapViewer
        │
        ▼
Zustand Store
  ├─ layers: LayerState[]
  ├─ activeLayerId
  ├─ stylePanel
  ├─ selection / hover / popups
  └─ interaction
        │
        ▼
LayerRenderer
  ├─ 同步线/面 DataSource
  ├─ 加载 GeoJSON 数据并回写 store
  ├─ 分发点图层到 PointRendererManager
  └─ 负责图层增删改同步
        │
        ├─ 线/面: GeoJsonDataSource / Entity
        └─ 点: PointRendererManager
               ├─ PointPrimitiveRenderer
               └─ BillboardRenderer
```

核心决策：

1. 逻辑图层与 Cesium 物理对象分离
2. 线/面使用 `DataSource`
3. 点使用 `scene.primitives`
4. GeoJSON 数据进入 store，渲染器只消费 store 中的标准化数据

## 3. 核心数据模型

当前渲染架构围绕 `LayerState` 和 `LayerStyle` 展开：

- `LayerState`
  - `id`, `name`, `type`, `geometryType`
  - `visible`, `opacity`
  - `style`
  - `sourceId`
  - `data`
  - `featureOverrides`

- `LayerStyle`
  - 通用：`color`, `width`, `opacity`
  - 点专属：`pointSize`, `pointSizeUnit`, `pointSymbol`, `pointRenderMode`
  - 点增强：`pointOutlineColor`, `pointOutlineWidth`, `pointRotation`, `pointImageUri`, `pointHeightOffset`
  - 标签：`label`

这套模型解决了两个问题：

1. 业务状态可序列化，可持久化，可由 UI 和 AI 共用
2. 渲染能力可以在不改业务模型的前提下继续扩展

## 4. 图层同步机制

`LayerRenderer` 是当前图层渲染总调度器，职责不是“画所有东西”，而是“同步 store 和 Cesium”。

当前同步流程：

1. 读取 `store.layers`
2. 从 `viewer.dataSources` 恢复或校正 DataSource 引用
3. 删除已失效的 DataSource 和 Point Renderer
4. 对每个图层判断：
   - 是否新增
   - 是否需要重载数据
   - 是否只需改显隐
   - 是否只需改样式
   - 是否需要切换点渲染模式

对应实现要点：

- 线/面图层：
  - `GeoJsonDataSource.load(...)`
  - 通过遍历 `entities` 更新样式

- 点图层：
  - 先拉取 GeoJSON
  - 回写 `store.updateLayerData`
  - 调用 `PointRendererManager.update(layer, viewer)`

这套机制的关键收益是“增量同步”而不是“整图重建”。

## 5. 点图层专用渲染架构

### 5.1 为什么点图层要单独设计

归纳自前面三份文档，原因有四个：

1. `GeoJsonDataSource` 默认点样式能力不够
2. 点图层更容易成为性能瓶颈
3. 点的表现方式天然多态：圆点、符号、图片、模型
4. 点图层后续编辑、hover、选择的交互路径与线面不同

### 5.2 PointRendererManager

`PointRendererManager` 是点图层渲染入口，职责是：

1. 根据 `pointRenderMode` 选择具体渲染器
2. 检测模式切换并销毁旧渲染器
3. 对外提供统一接口
4. 作为交互层和具体 renderer 之间的隔离层

当前支持：

- `point` -> `PointPrimitiveRenderer`
- `billboard` -> `BillboardRenderer`
- `model` -> 预留

### 5.3 PointPrimitiveRenderer

适用场景：

- 海量点
- 简单符号
- 对性能要求高

实现特征：

- 使用 `Cesium.PointPrimitiveCollection`
- 支持颜色、尺寸、轮廓、高度偏移
- 可选 `LabelCollection`
- 维护 `featureId -> primitive` 映射

### 5.4 BillboardRenderer

适用场景：

- 图片图标
- Canvas 符号
- 带旋转、原点控制的 POI

实现特征：

- 使用 `Cesium.BillboardCollection`
- 支持自定义图片和 Canvas 生成符号
- 支持 `sizeInMeters`
- 支持 `LabelCollection`
- 维护 `featureId -> billboard` 映射

## 6. 当前交互与渲染解耦的关键调整

这是近期实现里最重要的一次架构修正。

原问题：

1. 点图层 hover 逻辑曾被写进 `layer-renderer.tsx`
2. 直接修改原始 `PointPrimitive/Billboard`
3. 把 primitive 错当成 `Property` 使用，出现 `getValue is not a function`
4. 图层删除时点 renderer 没被纳入统一清理

现在的修正策略：

1. `LayerRenderer` 只做图层同步，不做 hover 交互
2. 新增 `point-renderer-registry.ts`，给交互层做 renderer 查询
3. 交互层通过 `findFeatureIdByPrimitive()` 反查要素
4. 点 hover 改为独立 overlay，而不修改原始 primitive

这意味着：

- 渲染负责“长期对象”
- 交互负责“短期反馈”
- 两者通过 registry 和 featureId 映射连接

这是当前图层渲染架构已经形成的稳定边界。

## 7. 图层删除与资源清理

图层删除目前有两层清理：

1. store 层
   - 删除 `layers`
   - 清理 `activeLayerId`
   - 清理 `stylePanel`
   - 清理 `selection / hover / popups`

2. 渲染层
   - 删除 `viewer.dataSources` 中的失效项
   - 销毁 `PointRendererManager`
   - 清理 registry

这里的经验教训很明确：

- 业务状态删除，不代表 Cesium 对象被删
- DataSource 路径和 Primitive 路径必须统一纳入清理
- 图层渲染系统必须把“资源生命周期”视为一等职责

## 8. 当前优势

现有图层渲染架构已经具备以下优点：

1. 线面与点分治，避免一套抽象硬扛所有图层
2. store 保持业务状态统一
3. 点图层支持多渲染模式切换
4. hover、选择、删除开始有清晰生命周期
5. 后续扩展 `model`、要素级 override、编辑态 overlay 的成本较低

## 9. 当前限制

虽然架构已经成型，但仍有几个已知限制：

1. 线/面样式更新仍依赖遍历 Entity
2. 图层重排序和 Cesium 实际渲染顺序还不是完全一一对应
3. 点图层点击选择仍未完全统一到 primitive 交互链路
4. `model` 渲染模式尚未落地
5. 要素级增量更新能力仍偏弱，部分更新依然靠整层重渲染

## 10. 后续建议

建议按下面顺序继续演进：

1. 完成点 primitive 点击选择与 popup 统一
2. 继续把 `LayerRenderer` 中残留的拾取逻辑迁出
3. 抽象线/面 overlay 高亮能力，和点 hover 机制统一
4. 评估图层重排序的渲染顺序控制方案
5. 为 `model` 渲染模式预留统一接口

## 11. 归档结论

这套图层渲染架构的本质不是“Cesium 的简单封装”，而是：

- 用 store 承接业务图层语义
- 用 `LayerRenderer` 承接同步职责
- 用专用 point renderer 解决点图层性能与表现力问题
- 用交互层 overlay 机制避免对原始渲染对象产生副作用

因此当前最合理的认知方式是：

1. 图层系统是业务层抽象
2. Cesium 只是底层渲染载体
3. 点图层已经是一个独立子系统
4. 后续所有编辑与交互能力，都应建立在这个边界之上继续扩展
